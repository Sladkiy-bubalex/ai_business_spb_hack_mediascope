# Архитектура решения — Mediascope Hackathon 2026

## Задача

Классификация поисковых запросов по трём атрибутам:

| Атрибут | Тип | Значения |
|---------|-----|----------|
| `TypeQuery` | int 0/1 | 1 = запрос о профессиональном видеоконтенте |
| `ContentType` | str | `фильм`, `сериал`, `мультфильм`, `мультсериал`, или `""` |
| `Title` | str | нормализованное название франшизы или `""` |

**Метрика лидерборда:**
```
combined_score = 0.35 × F2(TypeQuery) + 0.30 × macroF1(ContentType) + 0.35 × tokenF1(Title)
```

---

## Общая архитектура: трёхуровневый гибридный пайплайн

```
Входящий запрос (QueryText)
         │
         ▼
┌─────────────────────┐
│  1. Rule-based      │  ── уверен (confidence=1.0) ──────────────────┐
│     фильтр          │                                                │
└─────────────────────┘                                                │
         │ не уверен                                                   │
         ▼                                                             │
┌─────────────────────┐                                                │
│  2. Локальная       │  ── confidence > 0.85 ────────────────────────┤
│     ML-модель       │                                                │
└─────────────────────┘                                                │
         │ confidence ≤ 0.85                                           │
         ▼                                                             │
┌─────────────────────┐                                                │
│  3. YandexGPT API   │  ── всегда даёт ответ ────────────────────────┤
│  (новый контент,    │                                                │
│   сложные случаи)   │                                                │
└─────────────────────┘                                                │
                                                                       ▼
                                                         ┌────────────────────────┐
                                                         │  4. Confidence Router  │
                                                         │  confidence > 0.85 →   │
                                                         │    авто-разметка       │
                                                         │  confidence ≤ 0.85 →   │
                                                         │    очередь оператора   │
                                                         └────────────────────────┘
```

**Распределение нагрузки (оценка):**

| Уровень | Покрытие | Время на запрос |
|---------|----------|-----------------|
| Rule-based фильтр | ~35% | < 1 мс |
| Локальная ML-модель | ~45% | ~5 мс |
| YandexGPT API | ~20% | ~3–5 с |

**Итого 15 000 запросов: ~15 минут** (лимит — 4 часа)

---

## Два режима развёртывания

Решение существует в двух режимах, которые используют один и тот же код (`PredictionModel`), но отличаются тем, где выполняется тяжёлая классификация.

### Режим A — Хакатон (sandbox)

```
Песочница хакатона
  └── solution.py → PredictionModel.predict(df)
                         ├── Rule-based фильтр     (локально)
                         ├── Локальная ML-модель   (локально)
                         └── YandexGPT API         (сеть разрешена)
```

- Сеть открыта к Yandex Cloud / OpenAI / Anthropic
- Время: 10 минут на весь прогон
- Оценивается `combined_score` на тестовой выборке

### Режим B — Production on-premise (Airflow DAG)

```
Airflow DAG: mediascope_labeling
  │
  ├── Task: load_raw_queries
  │     Источник: внутренняя БД / S3-совместимое хранилище
  │     Выход: queries.csv (QueryText)
  │
  ├── Task: run_prediction  ← PredictionModel (этот же класс)
  │     Вход:  queries.csv
  │     Выход: labeled.csv (QueryText + TypeQuery + ContentType + Title + confidence)
  │     Где выполняется: on-premise GPU-сервер (RTX 4090)
  │     LLM: локальная модель (Qwen2.5-7B через ollama / vLLM)
  │     НЕТ внешних API — коммерческая тайна
  │
  ├── Task: route_by_confidence
  │     confidence >= 0.85 → auto_labeled.csv   (не требует оператора)
  │     confidence <  0.85 → review_queue.csv   (очередь оператора)
  │
  └── Task: write_results
        Сохранить в БД / передать в следующий DAG
```

**Ключевое отличие режима B:** YandexGPT API заменяется на **локальную LLM** (Qwen2.5-7B, Mistral-7B или аналог), запущенную через `ollama` или `vLLM` на том же GPU-сервере. Интерфейс `PredictionModel.predict()` не меняется — только переменная окружения `LLM_BACKEND=local`.

| Требование | Режим A | Режим B |
|------------|---------|---------|
| On-premise | Частично (API в облаке) | Полностью |
| Open-source | Да | Да |
| Коммерческая тайна | Данные уходят в Yandex Cloud | Данные не покидают контур |
| Переобучение | Ручное | Airflow DAG `retrain_model` |
| Скорость 15к | ~15 мин | ~30–60 мин (локальный LLM) |

## Компоненты системы

### Компонент 1 — Rule-based фильтр (`rules.py`)

Быстрая классификация на основе ключевых слов без обращения к ML или API.

**Логика:**
- Список стоп-слов → `TypeQuery=0`: `троллейбус`, `расписание`, `погода`, `спорт`, `новости`, `маршрут`, `курс валют`
- Список триггеров → `TypeQuery=1`: `смотреть`, `онлайн`, `скачать`, `сезон`, `серия`, `эпизод`, `hd`, `трейлер`, `аниме`
- Если совпадений нет → передать дальше

**Выход:** `TypeQuery` (0/1) + флаг `is_certain` (bool)

---

### Компонент 2 — Локальная ML-модель (`model.pkl`)

Обученный классификатор на тренировочных данных (14 000 строк).

**Архитектура модели:**
```
QueryText
    │
    ▼
TF-IDF (char n-grams 2–4, word n-grams 1–2)
    │
    ├──► LogisticRegression → TypeQuery (0/1) + confidence
    │
    └──► LogisticRegression → ContentType (фильм/сериал/мультфильм/мультсериал/"")
                              (только если TypeQuery=1)
```

Альтернатива для повышения качества: `rubert-tiny2` (~30 MB, работает на RTX 4090).

**Обучение:** `python train.py --data train.csv --output model.pkl`

**Выход:** `TypeQuery`, `ContentType`, `confidence` (0–1)

---

### Компонент 3 — Словарь тайтлов + fuzzy-matching (`titles_dict.json`)

Извлечение `Title` без обращения к LLM для известного контента.

**Логика:**
1. Построить словарь из `Title` колонки train.csv (все уникальные названия + алиасы)
2. Для запросов с `TypeQuery=1` — нечёткий поиск по словарю (`rapidfuzz`, порог 85%)
3. Если совпадение найдено → `Title` из словаря, `confidence` высокий
4. Если не найдено → новый контент → передать в YandexGPT

**Файл:** `titles_dict.json` (~несколько тысяч тайтлов)

---

### Компонент 4 — YandexGPT API (`solution.py`)

Вызывается только для ~20% запросов: новый контент + низкая уверенность ML.

**Параметры вызова:**
- Модель: `yandexgpt/latest`
- Температура: `0.0` (детерминированный вывод)
- Батч: 10 запросов за вызов
- Параллельность: 5 потоков одновременно
- Retry при 429: экспоненциальный backoff

**Промпт:** структурированный, с примерами few-shot, возвращает JSON-массив.

---

### Компонент 5 — Confidence Router

Маршрутизация результатов по уровню уверенности.

```python
if confidence >= 0.85:
    # Авто-разметка, оператор не нужен
    route = "auto"
else:
    # Низкая уверенность → очередь на ручную проверку
    route = "human_review"
```

**Бизнес-эффект:** операторы проверяют только ~15% запросов вместо 100%.

---

### Компонент 6 — Пайплайн переобучения (`train.py`)

Поддержка ежемесячного обновления модели на новых данных.

```
новые_данные.csv
      │
      ▼
   preprocessing
      │
      ▼
   train.py --data new_data.csv --model model.pkl --incremental
      │
      ▼
   model_v2.pkl  +  titles_dict_v2.json
      │
      ▼
   evaluate.py → combined_score
      │
      ▼
   bundle.zip → submit
```

**Режимы обучения:**
- `--incremental` — дообучение на новых данных (partial_fit)
- `--full` — полное переобучение с нуля

## Ключевые технические решения

| Проблема | Решение |
|----------|---------|
| Новый контент (не в train) | YandexGPT обрабатывает через zero-shot промпт |
| Опечатки и транслит | char n-grams в TF-IDF + fuzzy-matching |
| Скорость (15к за 4 ч) | Rule-based (35%) + ML (45%) + только 20% в API |
| Переобучение | `train.py --incremental` на новых данных ежемесячно |
| Уверенность | `predict_proba` от LogReg → Confidence Router |

### On-premise деплой (production)

```
GPU-сервер (RTX 4090 или аналог)
├── ollama run qwen2.5:7b          # локальная LLM
├── apache-airflow                 # оркестрация DAG
└── mediascope/
    ├── solution.py                # PredictionModel
    ├── model.pkl                  # локальные классификаторы
    └── titles_dict.json           # словарь тайтлов
```

Переключение между режимами через `.env`:
```
# Режим A (хакатон): внешний API
LLM_BACKEND=yandex
YANDEX_API_KEY=...

# Режим B (production): локальная модель
LLM_BACKEND=local
LOCAL_LLM_URL=http://localhost:11434  # ollama
```
